What Is XanaNode?

An explanation of XanaNode as a layered protocol, not a single viewer, website, repository, or graph visualization.

XanaNode is a layered protocol and working stack for knowledge substrates. It is not the website, not the graph alone, not the Hugo projection, and not a single repository.

At the bottom, the protocol defines the shared rules: canonical schemas, registries, artifact contracts, validation expectations, and governance conventions. Core reads and writes those protocol records. Workspace manages substrate folders, bundles, and federation intake. Studio and Mobile are authoring tools built on top of that substrate model. Hugo is one projection layer that renders the same underlying substrate as a public static site.

That layering matters because the knowledge should remain usable even when one projection changes. A substrate can be moved to another tool, shared as a .substrate bundle, mounted from a registry target, or rendered through a different projection without rewriting its meaning from scratch.

XanaNode is also an explicit answer to a three-part computing failure: document boundaries collapse knowledge structure, copied excerpts lose durable lineage, and local-first tools without a shared substrate protocol still do not federate cleanly. The protocol tries to solve that by keeping relationships first-class, making fragments addressable, preserving raw assets and provenance, and separating the substrate from any one projection layer.

The canonical XanaNode substrate is intentionally self-referential. It carries its own schemas, artifacts, sources, branding, project nodes, implementation links, and explanatory trails so the stack can demonstrate the thing it claims to provide.
